# Agentic Ruler Simulator (ARS) 架构设计

## 1. 项目愿景

打造一个基于大模型驱动的统治模拟器，融合《欧陆风云》《十字军之王》《维多利亚》《英白拉多：罗马》和全战系列里“国家机器、人物性格、派系利益”彼此牵制的体验。

首版目标不是做复杂地图或多人系统，而是做一个本地单人可玩的文字策略核心：玩家以统治者身份发布政策，系统让多个派系 Agent 独立反应，再由 Arbiter 汇总为叙事和数值变化。

当前版本已经引入“地缘战略层”：玩家不再只面对内廷派系，也会面对领土、邻国、动员、战争疲劳、世界紧张度和边境事件。

## 2. 系统架构：1 + N + World

### 2.1 The Arbiter（中枢协调者）

职责：

- 逻辑解析：将玩家的模糊文字指令转化为具体数值变化。
- 状态维护：管理资源、稳定、威望、合法性、危机和历史。
- 冲突调解：综合派系反馈，决定最终后果。
- 世界构建：开局时生成世界观、派系和初始危机。

输出：

- 更新后的 `GameState`。
- 派系反馈、数值变化和阶段性叙事总结。

### 2.2 Faction Agents（动态派系）

每个派系是一个独立 Agent，包含双层结构：

- 利益层：基于权力来源的长期诉求，例如武力、经济、信仰、土地、技术。
- 人格层：当前领袖的性格，例如顽固、贪婪、胆小、鲁莽、精明。

职责：

- 对玩家政策进行评价、游说、威胁、支持或讨价还价。
- 保留自己的记忆摘要，影响后续态度。
- 为 Arbiter 提供建议性的数值影响。

### 2.3 World Overlord（世界运行者）

职责：

- 维护邻国、边境、战争迷雾和世界紧张度。
- 每回合根据玩家国力、稳定度、邻国威胁触发外部事件。
- 解析扩张、外交、动员、建设、进攻等战略动作。
- 在攻击或战争指令中进行数值优先的战报结算。

### 2.4 External Liaison（外交接口）

职责：

- 将玩家对邻国的自由文本命令识别为外交、贸易、结盟、挑衅或威慑。
- 更新邻国关系、威胁度和可见情报。

### 2.5 War Room（战地总指挥）

职责：

- 当玩家发动攻击、动员或进入战争状态时激活。
- 基于军力、科技、动员、厌战、派系好感和随机波动进行战斗判定。
- 生成战报、损失、威望变化和后续边境压力。

## 4. 游戏循环

1. Input：玩家输入文本指令，例如“加税并安抚军队”。
2. Process：Arbiter 将指令广播给所有 Faction Agents。
3. Dialogue：派系根据利益和领袖人格做出反馈。
4. Action：Arbiter 综合反馈，计算最终 JSON 状态更新。
5. World：World Overlord 计算外部事件、战争结算和邻国反应。
6. Output：前端显示派系对话、数值变化、地缘看板、回合报告和当前局势描述。

## 5. 首版技术选型

- 后端：Python + FastAPI。
- Agent：LangChain Structured Output，缺少云端配置时使用本地 Mock 逻辑。
- 前端：Streamlit。
- 持久化：SQLite，每回合完整保存，可回滚。
- 模式：本地单人，不设计多人、账号、房间或同步。

## 6. 战略动作

- 扩张：消耗金钱和稳定，获得边疆省份，同时提高世界紧张度。
- 外交：消耗金钱，改善邻国关系并降低威胁。
- 动员：提升军力，但提高战争疲劳并压低稳定。
- 建设：在省份修筑要塞、粮仓、工坊、道路等设施。
- 进攻：基于军力、科技、动员、厌战、派系支持和随机波动生成战报。
